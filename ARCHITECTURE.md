# YAPI Plus 架构说明

## 🏗️ 整体架构

YAPI Plus 采用**前后端分离但可统一启动**的架构设计。

```
┌─────────────────────────────────────────┐
│          YAPI Plus 系统架构              │
├─────────────────────────────────────────┤
│                                         │
│  开发环境                                │
│  ┌──────────────┐    ┌──────────────┐  │
│  │   前端服务    │    │   后端服务    │  │
│  │  (端口4000)  │◄──►│  (端口3000)  │  │
│  │  VitePlus    │    │   Koa + API  │  │
│  └──────────────┘    └──────┬───────┘  │
│         ▲                   │          │
│         │                   ▼          │
│         │            ┌──────────┐     │
│         └────────────┤ MongoDB  │     │
│                      └──────────┘     │
│                                         │
│  生产环境                                │
│  ┌──────────────────────────────────┐  │
│  │        统一服务 (端口3000)        │  │
│  │  ┌────────────┐  ┌───────────┐  │  │
│  │  │  静态文件   │  │ API 服务  │  │  │
│  │  │ (前端构建)  │  │ (Koa)     │  │  │
│  │  └────────────┘  └─────┬─────┘  │  │
│  └────────────────────────┼────────┘  │
│                           ▼           │
│                    ┌──────────┐      │
│                    │ MongoDB  │      │
│                    └──────────┘      │
└─────────────────────────────────────────┘
```

---

## 🔄 两种运行模式

### 1. 开发模式（分离启动）

**架构特点**：

- 前端和后端**独立进程**运行
- 前端通过**代理**转发 API 请求到后端
- 支持**热更新**（HMR）
- 前后端可以**独立调试**

**启动方式**：

```bash
# 一键启动（推荐）
npm run dev

# 实际执行：
# 1. 启动后端: nodemon server/app.js dev -L
# 2. 启动前端: vp dev
```

**工作流程**：

```
浏览器请求
  ↓
前端开发服务器 (localhost:4000)
  ↓ (如果是 /api/* 请求)
代理转发
  ↓
后端 API 服务器 (127.0.0.1:3000)
  ↓
MongoDB
```

**优势**：

- ✅ 前端修改即时生效（HMR）
- ✅ 后端修改自动重启（nodemon）
- ✅ 前后端独立开发调试
- ✅ 完整的开发工具支持

---

### 2. 生产模式（合并启动）

**架构特点**：

- 前端**构建为静态文件**
- 后端**直接提供静态文件服务**
- **单一端口**访问
- **性能最优**

**启动方式**：

```bash
# 1. 构建前端
npm run build

# 2. 启动服务
npm start
```

**工作流程**：

```
浏览器请求
  ↓
后端服务器 (127.0.0.1:3000)
  ├─ /api/* → API 处理
  └─ /*    → 静态文件服务
       ↓
  static/prd/ (前端构建产物)
       ↓
  MongoDB (仅 API 请求)
```

**优势**：

- ✅ 单一进程，资源占用少
- ✅ 部署简单
- ✅ 性能最优
- ✅ 无需额外配置 Web 服务器

## 📦 前端构建产物

### 构建输出目录

```
static/prd/
├── index.html           # 入口 HTML
├── assets/
│   ├── index-xxx.js     # 主 JS 文件
│   ├── vendor-xxx.js    # 第三方库
│   └── style-xxx.css    # 样式文件
└── ...
```

### 后端如何提供静态文件

在 `server/app.js` 中：

```javascript
// 生产环境提供静态文件服务
const koaStatic = require("koa-static");

// 静态文件目录
app.use(koaStatic(path.join(__dirname, "../static/prd")));

// 所有非 API 请求返回 index.html
app.use(async (ctx, next) => {
  if (!ctx.path.startsWith("/api")) {
    ctx.type = "html";
    ctx.body = fs.readFileSync(path.join(__dirname, "../static/prd/index.html"));
  }
  await next();
});
```

## 🔌 代理配置

### 开发环境代理

在 `vite.config.js` 中配置：

```javascript
export default defineConfig({
  server: {
    port: 4000,
    proxy: {
      "/api": {
        target: "http://localhost:3000",
        changeOrigin: true,
      },
    },
  },
});
```

**工作原理**：

- 前端请求 `/api/user/login`
- VitePlus 代理到 `http://localhost:3000/api/user/login`
- 后端处理请求并返回结果
- 前端接收响应

## 🎯 为什么这样设计？

### 优势

1. **开发效率**
   - 前后端独立开发
   - 热更新快速迭代
   - 独立调试

2. **部署灵活**
   - 可以前后端分离部署
   - 也可以合并部署
   - 支持 CDN 加速

3. **性能优化**
   - 开发环境：快速热更新
   - 生产环境：单一进程，最优性能

4. **维护简单**
   - 一个命令启动所有服务
   - 无需复杂的配置
   - 适合各种规模团队

## 🔧 高级配置

### 自定义端口

**修改前端端口**（vite.config.js）：

```javascript
server: {
  port: 4001,  // 改为 4001
}
```

**修改后端端口**（config.json）：

```json
{
  "port": 3001 // 改为 3001
}
```

### 分离部署

如果前端和后端部署在不同服务器：

**前端配置**（.env.production）：

```env
VITE_API_URL=https://api.yourdomain.com
```

**后端配置**：

- 配置 CORS 允许前端域名
- 提供 API 服务
